# Поручения сотрудникам («умный посредник») — план и журнал

Документ — **память проекта** по этой фиче: полный замысел и что уже сделано после каждого шага.

---

## Цель

Руководитель формулирует поручение **своими словами** → бот **понимает**, **ставит на сотрудника**, **пишет в tasks**, **контролирует чеклист** (в последующих фазах). Личные напоминания — **в ЛС** сотруднику (после `/start` и привязки в `employees`).

---

## Роли и каналы

| Роль | Где пишет | Задача |
|------|-----------|--------|
| Руководитель | Общий чат или личка боту | Даёт поручения, смотрит статус |
| Сотрудник | **Личка боту** | Получает поручение, шлёт фото/текст «сделано» |
| Бот | Оба канала | Парсит запрос, пишет в Sheets, отвечает |

**Правило:** поручение и ответ по нему — **в ЛС**; в общий чат — краткое подтверждение при необходимости.

---

## Идентификация сотрудника

- Резолв по **`employees.name`** (без учёта регистра).
- При неоднозначности — **уточняющий вопрос** руководителю, без создания задачи.
- Без **`telegram_user_id`** — не слать ЛС; объяснить: нужен **`/start`** (или строка в таблице с `@username` + повторный `/start`).

---

## Модель данных (эволюция)

**Сейчас (фазы 1–2):** лист **`tasks`**: `task_id`, `assigned_to`, `title`, `due_date`, `notes`, `status`…  
В `notes` может быть блок **`__DELEGATION_JSON__`** с `{"delegation": {"version": 1, "checklist": [...]}}` (создаётся через **`delegate_private_reminder`** + `checklist_items`).

**Позже:** отдельный лист **`delegations`** или расширение статусов.

---

## Память между чатами (аналог Mira+, уровень 2)

**Цель:** в отличие от Mira, бот «помнит», что происходило **во всех чатах**, а не только в текущем.

| Уровень | Что | Статус |
|---------|-----|--------|
| 1 | Общие **Sheets** (tasks, schedule, events) + **`memory_facts`** на все чаты | Реализовано с начала |
| 2 | **Сквозная история:** последние N реплик из **других** чатов в system prompt | **Реализовано** |
| 3 (опц.) | Авто-сводки / Notebook LM / полный архив переписки | Не в плане MVP |

**Реализация уровня 2:**
- `get_recent_history_other_chats(exclude_chat_id, limit)` — `src/google/sheets.py`
- `build_cross_chat_system_section` — `src/agent/cross_chat_memory.py`
- `call_agent` → блок **«Недавно в других чатах»** в system instruction
- Подписи чатов из листа **`chats`** (`chat_name`)
- Env: **`MEMORY_CROSS_CHAT_ENABLED`** (default true), **`MEMORY_CROSS_CHAT_LIMIT`** (default 20), **`MEMORY_CROSS_CHAT_MAX_CHARS`** (default 400 на сообщение)

**Ограничения:** не вся переписка за месяц — только последние N строк из `memory_history` других чатов; для долгосрочного — **`save_fact`** и задачи в таблице.

## Статусы жизненного цикла (целевые)

`draft` → `sent` → `awaiting_proof` → `review` → `done` / `blocked`  

Сейчас в таблице исторически используются `pending` / `done`; расширение — по мере фаз.

---

## Связь «ответ в ЛС = задача X»

**Фаза 3 (реализовано):** в личке обработчики **`text`** / **`photo`** смотрят **`reply_to_message`**: если это сообщение **от бота**, из текста/подписи извлекается **UUID** (`src/bot/delegation_reply.py`). ID передаётся в префикс контекста (`reply_task_id`). Tool **`assert_task_for_telegram_user`** проверяет, что задача назначена этому **telegram_user_id**, перед **`complete_task`**.

**Fallback:** несколько открытых поручений без реплая — см. фазу 2 (**get_open_tasks_for_telegram_user**).

## Чеклист и фото (фазы 2+)

- Погода (Москва) — отдельный tool при необходимости.
- Vision по фото + структурированный промпт по пунктам.
- Результат писать в **`notes`** задачи или отдельное поле.

## Фазы внедрения

### Фаза 1 — поручение + учёт + ЛС с task_id

- Один tool **`delegate_private_reminder`**: `create_task` + **`send_dm_to_employee`**; в ЛС добавляется **`task_id`** и просьба ответить реплаем.
- Промпт агента: когда руководитель просит передать поручение в личку — вызывать этот tool.
- **Журнал:** см. низ документа.

### Фаза 2 — чеклист в notes + уточнение «к какой задаче» без реплая

- Параметр **`checklist_items`** у **`delegate_private_reminder`** → JSON в **`notes`** после маркера **`__DELEGATION_JSON__`**.
- Tool **`get_open_tasks_for_telegram_user(telegram_user_id)`** — открытые задачи + чеклист из notes; для сценария «готово» без реплая.
- В **личке** к тексту пользователя добавляется префикс с `telegram_user_id` (`src/bot/private_agent_message.py`), чтобы модель вызывала tool с верным id.
- Inline-кнопки выбора задачи — **не** в этой итерации (достаточно списка + явный `task_id`).

### Фаза 3 — обработчик ЛС: `reply_to_message` → `task_id`

- **`src/bot/delegation_reply.py`**: из сообщения бота (реплай) — первый UUID в text/caption.
- **`text.py` / `photo.py`**: в личке передают **`reply_task_id`** в **`with_private_telegram_context`**.
- **`assert_task_for_telegram_user`** в **`tools.py`** + регистрация в **`client.py`**; промпт требует вызывать перед **`complete_task`** при отчёте реплаем.

### Фаза 4 — напоминания, эскалации, опционально общий чат

- **`src/scheduler/task_reminders.py`**: в составе **`run_automations`** (каждые 5 мин) — задачи **pending** с **due_date ≤ сегодня** (TZ приложения); не более **одного ЛС в день** на задачу (маркер **`__TASK_REMINDER_DATE__:`** в **notes**); инкремент **`reminder_count`** до **`TASK_REMINDER_MAX_SENDS`** (env, по умолчанию 3).
- После **`TASK_REMINDER_ESCALATE_AFTER`** успешных напоминаний (по умолчанию 2) — **одно** сообщение в **первый активный** чат из **chats** (маркер **`__TASK_GROUP_ESCALATION__:v1`** в notes).
- Tool **`send_brief_to_primary_work_chat`** — краткий текст в тот же чат; cooldown **1 час** через **`memory_facts`** (`employee=_system_group_notice_cooldown`).
- Параметры: **`config.py`** / env **`TASK_REMINDER_MAX_SENDS`**, **`TASK_REMINDER_ESCALATE_AFTER`**.

### Фаза 5 — контроль чеклиста, статусы, UX

**Статус:** реализовано (2026-05-16).

Цель фазы: закрыть обещание из «Цель» — бот **контролирует чеклист**, а не только хранит его; руководитель видит **review** перед финальным **done**.

| Шаг | Что сделать | Критерий готовности |
|-----|-------------|---------------------|
| **5.1** | Расширить статусы в **`tasks.status`**: минимум `pending`, `awaiting_proof`, `review`, `done`, `blocked`. Обновить **`complete_task`**, фильтры открытых задач, напоминания (не слать по `done`/`blocked`). | В таблице и в ответах бота различаются «ждём отчёт» / «на проверке» / «закрыто». |
| **5.2** | Tool или handler **`submit_task_proof`**: фото + `task_id` (из реплая или явно) → Vision по чеклисту из **`__DELEGATION_JSON__`** → структурированный результат (пункт: ok / не видно / нет). | В **`notes`** дописывается блок отчёта с датой; статус → `review` или `awaiting_proof`. |
| **5.3** | Правила агента: при неполном чеклисте **не** вызывать **`complete_task`**; предложить дослать фото или уточнить; при полном — `review` и уведомление руководителю (ЛС или кратко в чат — отдельное решение). | Сотрудник не может «закрыть» задачу одним словом, если чеклист не пройден (если чеклист был задан). |
| **5.4** | Tool **`approve_task`** / **`reject_task_proof`** для руководителя: из `review` → `done` или обратно в `awaiting_proof` с комментарием в **`notes`**. | Руководитель одобряет/возвращает на доработку из чата. |
| **5.5** | Inline-кнопки в личке при **2+** открытых поручениях: callback → тот же контекст, что **`reply_task_id`** (без ручного копирования UUID). | Сотрудник выбирает задачу кнопкой, не путает поручения. |
| **5.6** | (Опционально) Парсинг **`task_id`** из текста сообщения (UUID), если реплай/кнопки не использованы. | «Готово по задаче &lt;uuid&gt;» привязывается без реплая. |
| **5.7** | (Опционально) Напоминания фазы 4+: выбор чата эскалации не только «первый в chats»; отдельное ЛС руководителю при эскалации. | Эскалация попадает в нужный чат/человека. |

**Порядок внедрения:** 5.1 → 5.2 → 5.3 → 5.4 → 5.5; 5.6–5.7 по запросу клиента.

**Не входит в фазу 5 (отдельный backlog):** лист **`delegations`**, Google Tasks sync, мультитенантность, tool погоды.

## Приёмочное тестирование (фазы 1–4) — для клиента

Перед стартом фазы 5 клиент проверяет уже выкатанный функционал. Чек-лист для передачи клиенту — в конце этого файла (раздел используется как эталон; текст для Telegram/почты — у разработчика).

**Подготовка (один раз):**
- Сотрудник в листе **`employees`**: **name** как в разговоре, при необходимости **@username**.
- Сотрудник нажал **`/start`** у бота в **личке** (появился **telegram_user_id**).
- В **`chats`** есть рабочий чат с **`active=true`** (для эскалаций и **`send_brief_to_primary_work_chat`**).
- Бот запущен на сервере; планировщик работает (напоминания раз в ~5 мин).

| № | Кто | Действие | Ожидаемый результат |
|---|-----|----------|---------------------|
| 1 | Руководитель | В общем чате или личке: «Передай Ивану: до 18:00 проверить витрину, напомни в личку» | Бот подтверждает; в **`tasks`** новая строка; Ивану в **ЛС** сообщение с **ID задачи** и просьбой ответить **реплаем**. |
| 2 | Руководитель | То же с чеклистом: «… и пункты: пол, витрина, вывеска» | В **`notes`** задачи есть **`__DELEGATION_JSON__`** с тремя пунктами (или бот явно перечислил чеклист). |
| 3 | Сотрудник | **Реплаем** на сообщение бота: «Сделал» или фото | Бот принимает отчёт; задача в **`tasks`** → **done** (или понятный ответ, если не уверен). |
| 4 | Сотрудник | «Готово» **без реплая**, при **одной** открытой задаче | Бот закрывает или уточняет однозначно. |
| 5 | Сотрудник | «Готово» **без реплая**, при **двух** открытых поручениях | Бот показывает список с **task_id** и просит реплай или указать ID. |
| 6 | Руководитель | «Какие задачи у Марии?» / статус поручения | Бот отвечает по данным из **`tasks`**, без выдумок. |
| 7 | Система | Задача с **due_date = сегодня**, статус не done | В течение дня сотруднику приходит **одно** напоминание в ЛС (не дубли в тот же день). |
| 8 | Система | Задача просрочена, **reminder_count** достиг порога (по умолчанию после 2-го напоминания) | **Одно** сообщение об эскалации в **рабочий чат** из **`chats`**. |
| 9 | Руководитель | «Напиши в общий чат: через 10 минут планёрка» (кратко) | Сообщение в первом активном чате; повтор сразу — отказ из‑за лимита **1 раз в час**. |
| 10 | Руководитель | Поручение сотруднику **без /start** | Задача в таблице может создаться; в ответе — что ЛС не доставлено, нужен **/start**. |

**Что зафиксировать клиенту при баге:** скрин, время (МСК), чат (личка/группа), текст запроса, **task_id** из ЛС, ожидание vs факт.

## Риски

- Несколько открытых поручений без реплая → путаница.
- Vision ошибается → нужен ручной контроль руководителя.
- Лимиты длины сообщений Telegram.
